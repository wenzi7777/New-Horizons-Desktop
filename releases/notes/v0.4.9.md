# New Horizons Desktop v0.4.9

## Features

### Stale device offline detection

- Added a backend stale-device watchdog (`NewHorizonsService`): every 15 s it
  inspects devices that are still flagged `gateway_connected` and, if no
  presence message has been recorded for 30 s (`STALE_DEVICE_TIMEOUT_SEC`),
  marks the device offline and emits a `device_update` event. With the gateway
  pushing device presence every 5 s, this gives ~6 missed cycles of tolerance
  before a device is declared gone.
- Because 30 s exceeds the frontend's 15 s reconnect grace, a stale device
  transitions directly from **online** to **offline** (no confusing "reconnecting"
  blink for a device the backend has concluded is truly absent).

### Launchpad folder preview redesign

- Replaced the 2×2 device-code chip grid and caption on folder preview cards
  with a single-column list: a colored status dot, the device display name, and
  its mode, up to 6 items, with a `+N more` overflow row.
- Empty folders render at reduced opacity and stay non-interactive, matching
  the disabled state.

## Fixes

### Stale check never marked gateway-relayed devices offline

- The frontend (`device.ts`) treats a device as connected when **either**
  `device.gateway_connected === true` **or** the embedded `gateway_state.connected
  === true` is true. The original stale check only cleared the top-level
  `gateway_connected`, leaving `gateway_state.connected` untouched — so
  gateway-relayed devices (the feature's main target) never appeared offline.
  The check now updates the embedded `gateway_state.connected` in lockstep with
  the top-level flag.

### Stale timer leaked / rescheduled after stop()

- The watchdog rescheduled itself unconditionally, so a `_check_stale_devices`
  callback that ran concurrently with `stop()` could arm a fresh timer after
  `stop()` had already nulled and cancelled it — keeping the service's timer
  thread alive indefinitely. Introduced a `_stopped` flag: `stop()` sets it and
  `_schedule_stale_check` refuses to arm (and `_check_stale_devices` bails out)
  while stopped.

### Held-lock event emission and timestamp parsing

- Moved `_emit_event` calls out of the `self._lock` block to match every other
  call site in the service (listeners now run without the outer RLock held).
- Replaced the raw `datetime.fromisoformat(...).timestamp()` with the existing
  `_parse_iso_datetime` helper (handles `'Z'` suffixes and naive datetimes
  consistently) and removed an unused `now_iso` variable.

## Verification

- Added regression coverage in `tests/test_backend_gateway.py`: a stale
  gateway device is marked offline (both `gateway_connected` and
  `gateway_state.connected` cleared) with a `device_update` event; a fresh
  device stays connected; after `stop()` the timer is not re-armed and the
  stale check no longer mutates device state.
- Updated the launchpad markup snapshot assertions in
 `tests/test_app_auth_static.py` to match the new preview list classes.
- Verified the full backend test suite (`python3 -m unittest discover -s tests`)
  and the frontend production build (`tsc -b && vite build`).
